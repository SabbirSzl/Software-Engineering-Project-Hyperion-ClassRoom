1. Introduction

1.1 Purpose

The software discussed in this SRS is a virtual platform that simplifies the interactions between students and teachers. The software will facilitate the distribution of class tasks and offer student activity monitoring during interaction time.

1.2 Document Conventions

We use the standard HTML convention to create headers and subheaders for different sections. In section 4, we used tables to visualize expanded use cases. The web page is printable from the browser and uses the standard IEEE convention for software requirements specification (SRS) document.

1.3 Intended Audience and Reading Suggestion

Teachers, students, and invigilator who wants a robust assignment and task distribution system with monitoring features are the primary audience for this software.

1.4 Product Scope

The software is a virtual platform for student and teacher interacion. The software will give additional advantage of monitoring tools that can be used by a teacher. It can also provide analytics and reports from these tools that a teacher can use to ensure fair grading. The rpoduct can be used by teachers and students from different instituions to provide a seamless experience for both students and teachers.

2. Overall Description

2.1 Product Perspective

Due to the global pandemic, the demand for an online platform for student-teacher interaction has grown. The need for a virtual platform that can substitute the traditional classroom and create a fast distribution system has inspired us to take this project. Softwares like google classroom already offer such a platform but, our software aims to offer a monitoring tool that can ease the job of invigilators and teachers and provide them with analytics that will aid them to ensure fair grading.

2.2. Product Functions

The major system functions are given below:

  • Users create account using their email or sign in using gmail.
  • Teacher can create a classroom for interaction.
  • Teacher and students can post on classroom and comment on posts
  • Monitor student whether their eye is open or not while reading documents.
  • Use facial recognition to cross-check student identity
  • Monitor student engagement in particular topic of a task.
  • Deliver all monitoring report to teacher.

2.3. User Classes and Characteristics

The following are the user classes in our software:

  1. System Admin
    • Has system database access.
    • Can update or modify a user.
  2. Teacher
    • Create a classroom
    • Create Posts
    • Comment on Posts
    • Use monitoring tools.
    • Upload questions.
    • Grade students.
  3. Student
    • Join a class
    • Create Post
    • Comment on posts
    • View documents
    • View grades

User class "Teacher" is the most important and privileged in this system. This user class can interact with teacher and use speicfic monitoring tools that are exclusive only to the teacher. Students are less privelged and has limited accessibility. The system admin is a special user class that is reserved for system administrators who can revoke or modify user accounts.

2.4. Operating Environments

Following are the system requirements for the software:

    Computer
  • Hardware: x64/x34
  • Operating System: Windows/Linux

    • Android
  • Hardware: ARM architecture
  • Operating System: Android 7.0+

2.5. Design and Implementation Constraints

List of Constraints are given here:
  • User facial snapshots won't be stored for privacy reason
  • User must be notified before accessing their camera
  • User credentials must be checked every session
  • User will be notified of the tools that will be used for monitoring

2.6 User Documentation

The software is designed to be user friendly. The software will contain intuitive features that the user can learn quickly. There will also be an FAQ page for easy solutions.

2.7 Assumptions and Dependencies

The software will depend on third party api from Microsoft or Google for face recognition, eye tracking or student engagement. The limitations of the API can limit the performance of the software

3. External Interface Requirements

3.1 User Interfaces

The software will contain a login page where signup option will also be available. User can also choose to signup using Gmail. After login/signup user will see dashboard according to their user class. The "teacher" user then can create a class and create and comment posts. The "student" user can use their interface to join a class. The teacher can use monitoring tools from dashboard interface. The teacher can also receive report and analytics from dashboard. The web app and android app will have similar interface.

3.2 Hardware Interfaces

Computer:

  • Ram: 1 GB
  • OS: windows/linux
  • Webcam: Minimum 4mp
Android:
  • OS: Android 7.0+
  • Front camera: Minimum 4mp

3.3 Software Interfaces

The software needs a database. So we will be using good database management system software. Also, we will be integrating third party api in our software like MS Azure or Google vision API. We will use any modern browser with support for javascript, html5 and css.
For android phone, we will use native andriod app development kit and kotlin.

3.4 Communications Interfaces

Communication between system and user will occur using api that will facilitate email notifications. Face recognition and eye track tool will notify user before action.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

  • The system should be able to access the webcam to process facial recognition, track eyes in real time with zero delay and render documents to generate a report based on users activity.
  • All documents uploaded by the teacher should not be downloadable.
  • System should not take more than 5 seconds to appear when a teacher assign a task.
  • User must have good internet connection to provide better performance.

5.2 Safety Requirements

  • User is responsible for content upload.
  • User may not post something derogatory.

5.3 Security Requirements

  • User is responsible for maintaing his/her account.
  • User credentials should no be shared with anyone.
  • All users should be authenticated before granting access.
  • Student should be enrolled in a course before accessing its materials and attempting its assignments and exams.

5.4 Software Quality Attributes

  • As some features are in real time so, performance depends on the availibility of bandwidth.
  • Accurate and precise process must be performed by the system to avoid problems.
  • GUI of the system will be simple so everyone can easily understand.
  • The design of the Classroom system should allow to run on all widely used browsers.

5.5 Business Rules

  • Administrator have a complete control to the system to edit, add/remove a user. Ensures valid information.
  • Teacher will be able to remove any student and submit grade and edit any data.
  • Students will be able to join a class, post to class and see all other activities in class.

6. Other Requirements

This system requies a database and a server to store the data.
